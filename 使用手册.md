## 2. 今天客户运营还没做 — 怎么告诉智能体？

### 场景 1：8:00 cron 没跑或失败了，要补跑

**你说**（任选一种）：

- "今天 4 个客户的运营都没做，跑一遍"
- "全巡检"
- "/全巡检"

智能体动作：
1. 跑 daily-cron 4 客户并行（约 14-18 分钟）
2. 输出企微 6 段卡片（1 头部 + 1 前置 + 4 客户）
3. 写 4 个客户日报到各自 `客户/<X>/website/docs/client-briefing-<日期>.md`
4. 写 commit 到各客户站

### 场景 2：只想跑某一个客户

- "只跑 demo-c 的运营"
- "/巡检 demo-c"

智能体跑单客户，约 7-10 分钟。

### 场景 3：要写博客但不知道写啥

- "给 demo-b 出一篇博客"（零参数，智能体自动选题）
- "/博客 demo-b"

智能体动作：
1. 跑 topic-pool（找 GSC Gap + 竞品没写的）
2. 选潜力分最高的主题
3. 走 12 阶段 content-production
4. 6 语种翻译 + 部署 + IndexNow + commit

### 场景 4：要写指定主题博客

- "给 demo-c 写一篇关于 EPS 价格 2026 趋势的博客"
- "/博客 demo-c EPS price 2026 trends"

### 场景 5：客户网站出问题（紧急）

- "demo-a 网站打不开了，紧急修一下"
- "/救火 demo-a 网站不可达"

智能体动作：诊断 → 修复 → 验证 → 复盘 → 防复发规则更新（5 步硬闭环）。

### 场景 6：想看某客户当前状态

- "demo-c 现在怎么样"
- "看 demo-b 最近 7 天 GSC 数据"
- "/周检 demo-c"

### 场景 7：让 4 客户都跑 CTR 优化

- "4 客户都跑一遍 CTR 优化"
- "/CTR demo-c" + "/CTR demo-b" + ...（一次一个）

## 3. 怎么判断智能体真在工作（不是糊弄你）

**4 个验证信号**（任一缺失说明出问题）：

1. **企微群推送** — 6 段独立卡片，每段有 `[N/4 客户名]` 标号 + "客户日报质量 X/100"
2. **客户站 git log** — 今日有新 commit（看 4 客户的 `git log --since="today"`）
3. **客户日报新生成** — `客户/<X>/website/docs/client-briefing-<今日>.md` 修改时间 = 今天
4. **pending-tasks 闭环** — `客户/<X>/website/docs/pending-tasks.md` 有今日 closed 条目

如果 4 个都齐 = 智能体真干活；如果只有 1（推送）但 2/3/4 缺 = 假繁荣。

## 4. 评分系统（防退化）

每客户日报推送时自动附 **客户日报质量 X/100 ✅/⚠️/🔴**：

- **≥ 80 ✅ 通过**（可直接转发客户）
- **60-79 ⚠️ 警告**（建议人工微调，但能发）
- **< 60 🔴 待人工审**（千万别发，先改）

**评分维度**（各 25 分）：
- 数据完整度（GSC 7d / GA4 24h / 索引 / SSL）
- 行动可执行性（≥ 3 项今日执行 + 每项含"预期效果"）
- 客户语言（第二人称 ≥ 5 次 / 内部术语必脱壳）
- 排期清晰度（明天 / 本周 / 接下来）

**发酵期客户特殊**：≤ 30 天上线（Demo-D client-A / demo-a client-B2）GSC/GA4 数据少，标"发酵期"自动豁免。

## 7. 故障排查

### 智能体没收到企微推送

```bash
# 1. 检查 pm2 进程
pm2 list

# 2. 看 daily-error.log
tail -50 ${WORKSPACE_ROOT}/mcp-servers/wecom-bot/logs/daily-error.log

# 3. 健康监控日志
tail ${USER_HOME}/.local/share/web-ops/pm2-health-$(date +%Y%m%d).log

# 4. 手动跑一次
cd ${WORKSPACE_ROOT}/mcp-servers/wecom-bot && node daily-cron.mjs
```

### 某客户失败（其他 OK）

```bash
ls -t ${WORKSPACE_ROOT}/mcp-servers/wecom-bot/logs/claude-<client-id>-*.log | head -1
# 看 dump 文件 stdout/stderr 找原因
```

常见原因：
- "Reached max turns" → 该客户 prompt 太重，加 phase 字段或 maxTurns
- "exceed max length 4096" → 推送超长（v10.2 已改分段，应不再出现）
- "You've hit your limit" → API 限额，等几小时

### 客户日报评分低（< 60）

智能体已自动标"待人工审"，看具体扣分项：
```bash
node ${WORKSPACE_ROOT}/智能体/运营/网站运营-web-ops/scripts/briefing-quality-scorer.mjs --client demo-c --date 2026-04-29
```

输出会列出哪些维度缺失，针对性补。

## 8. 我（运营人员）的 4 个角色定位

| 角色 | 啥时候用 | 例 |
|---|---|---|
| **派活的** | 大方向需求 / 紧急任务 | "今天给 4 客户都跑 CTR" / "demo-a 网站打不开了救火" |
| **决策者** | 智能体推 B/C 级动作给我审 | 简报里 "🟡 需您决定"段，看完决定做不做 |
| **资料供应商** | 客户配合 / 账号注册 / 素材 | HARO 注册 / Clarity 埋点 / 客户 expertise pack |
| **审核员** | 评分 < 60 红字 / 简报转发前 | 看 quality gate 标签，低分的手动微调 |

**我不该做的事**（智能体已自主，做了浪费时间）：
- ❌ 写 Title/Desc / 改 H2 / 选 FAQ 问题（智能体专业拍板）
- ❌ 跑日常巡检（cron 自动）
- ❌ 部署 / IndexNow / git commit（智能体 zero-touch A 级自主）
- ❌ 翻译 6 语种（content-production 阶段 10 自动）
