# 每周检查技能

> 触发指令：`周检查` / `[客户名] 周检查`  
> 执行频率：每周一次（建议周一）  
> 人工确认：无需确认。发现严重问题时报告并建议触发hotfix

---

## 模式一：全局周检（`周检查`）

### MCP 调用链

```
Step 1: 基础健康（并行）
        ┌→ site-monitor.check_all_sites()
        ├→ site-monitor.check_ssl_expiry()
        └→ client-manager.list_clients(active)

Step 2: 逐客户SEO检测（遍历活跃客户，并行执行）
        → seo-checker.check_seo(域名, skip_pagespeed=true)

Step 3: 内容状态
        → content-tracker.content_summary()

Step 4: 汇总输出周报
```

---

## 模式二：单客户深度周检（`[客户名] 周检查`）

### MCP 调用链

```
Step 1: 加载客户
        → client-manager.get_client(client_id)

Step 2: 全面检测（并行）
        ┌→ site-monitor.check_site(域名)
        ├→ seo-checker.check_seo(域名)
        ├→ seo-checker.check_pagespeed(域名)
        ├→ seo-checker.check_structured_data(域名)
        ├→ seo-checker.batch_check([关键页面列表])
        └→ content-tracker.list_content(client_id)

Step 3: 站点地图分析
        → fetch(域名/sitemap-index.xml 或 域名/sitemap.xml)
        → 提取所有URL，与实际页面对照
        → 识别：sitemap中有但404的URL / 正常页面但不在sitemap中的

Step 4: robots.txt 审计
        → fetch(域名/robots.txt)
        → 检查是否有误屏蔽的正常页面

Step 5: 竞品快速扫描（如客户有竞品列表）
        → fetch(主要竞品博客页) × 1-2个
        → 检查竞品是否有新内容发布

Step 6: 内链健康度
        → 抽查5个页面的内链情况
        → 识别孤岛页面（无入链的页面）
        → 检查断链（链接指向404的内链）

Step 7: 索引健康度评估（search-analytics MCP 自动化）
        → search-analytics.gsc_index_status(site=域名, check_all_en=true)
        → 真实索引率 = 已索引页面 / sitemap总页面
        → search-analytics.gsc_crawl_errors(site=域名) → 新增404/403/重定向错误
        → 与上周对比：新索引/掉索引/新错误
        → 索引率<50% → 🔴 告警
        → 新增404 → 立即标记修复

Step 8: ⚡ 立即修复（检查→修复 闭环，不允许跳过）
        → 遍历Step 2-7发现的所有问题
        → 分类：
          🔴 当场可修（<30分钟）→ 立即触发 hotfix 修复 → 构建部署 → 回验
          🟡 本周内修（需较多时间）→ 制定具体修复计划+时间
          ⚪ 非技术问题（需客户配合）→ 明确告知客户需要做什么

        → 修复执行后，重新检测确认问题消失
        → 如果有构建部署，必须回验线上状态

Step 9: 输出周报（报告是修复记录，不是待办清单）
        → 保存到 客户-XX/数据/YYYY-WXX_周检报告.md
        → 报告结构：
          - 健康评分
          - ✅ 已修复的问题（含修复前/后对比）
          - 🟡 本周内待修的问题（含计划）
          - ⚪ 需客户配合的事项
          - 🟢 状态良好的维度
        → client-manager.add_timeline(id, "周检: XX分 | 修复X项 | 待修X项")
```

### GSC数据使用规范（必读）

> 来源：客户B实战教训——向客户报"排名第1"，客户搜不到，信任受损。

1. **过滤假关键词**：query 中包含 `-site:` 或引号+排除条件的，必须从报告中剔除
2. **排名用"平均排名"**：position 6.5 不代表稳定在第6名，报告中标注"GSC平均排名"
3. **禁止夸大表述**：禁用"登顶""排名第1""已进入首页"。用"GSC显示平均排名X.X"
4. **区分词类**：长尾词（4+单词）排名预期与核心商业词（1-2词）完全不同，不混谈
5. **客户可验证声明**：报告末尾加注"排名因搜索地区、语言、设备不同而异，以GSC数据为准"

## 健康评分标准

### 可用性（20分）
- HTTP 200：+10
- 响应<1s：+5 / 响应1-3s：+3 / 响应>3s：0
- SSL>60天：+5 / SSL 30-60天：+3 / SSL<30天：0

### SEO基础（25分）
- Title正确：+5
- Description正确：+5
- H1存在且唯一：+5
- OG标签完整：+5
- Canonical正确：+5

### 结构化数据（15分）
- Organization Schema：+5
- Product/Article Schema：+5
- FAQ/BreadcrumbList Schema：+5

### 索引健康（20分）— v9.0用GSC真实数据
- 索引率>70%：+5 / 50-70%：+3 / <50%：0（数据来源：gsc_index_status）
- 无新增404：+5 / 有新增404但已修：+3 / 未修：0（数据来源：gsc_crawl_errors）
- 无掉索引页面：+5 / 掉索引<3个：+3 / ≥3个：0（数据来源：gsc_index_changes）
- robots.txt无误屏蔽：+5

### 内链质量（10分）
- 无孤岛页面：+5
- 无断链：+5

### 内容新鲜度（10分）
- 7天内有新发布：+10
- 14天内：+7
- 30天内：+4
- >30天：0

## 异常处理

| 发现 | 动作 |
|------|------|
| 🔴 紧急问题 | 立即建议执行 hotfix |
| 健康评分<60 | 标记为重点关注客户 |
| 内容>30天未更新 | 提醒安排内容生产 |
| 竞品有重大动作 | 建议调整内容策略 |
| 评分连续2周下降 | 深度诊断根因 |
