# 竞品监控与分析技能

> 触发指令：`[客户名] 竞品调研`（深度模式）/ 在 data-analysis 和 daily-ops 中自动调用（快速模式）  
> 执行频率：深度模式每月1次（Day 1-2）；快速模式每日巡检自动执行  
> 人工确认：无需确认，输出简报供选题参考  
> 数据来源：client-manager（竞品列表+notes字段） + fetch（页面抓取） + seo-checker（技术对比） + content-tracker（我方内容）

---

## MCP 调用链：深度竞品分析（`[客户名] 竞品调研`）

```
Step 1: 加载客户+竞品列表
        → client-manager.get_client(client_id)
        → 从 notes 字段提取竞品列表（域名+名称）
        → 如竞品列表为空 → 提示先用 update_client 添加竞品

Step 2: 竞品网站技术检测（并行）
        → 对每个竞品域名：
          ┌→ site-monitor.check_site(竞品域名)        // 可用性+响应时间
          ├→ seo-checker.check_seo(竞品域名)           // SEO基础
          └→ seo-checker.check_structured_data(竞品域名) // Schema
        → 输出：每个竞品的技术评分对比

Step 3: 竞品内容抓取（并行，每个竞品2-3页）
        → fetch(竞品1/blog) → 提取：最新5篇文章标题+日期+主题分类
        → fetch(竞品2/blog) → 同上
        → fetch(竞品N/blog) → 同上
        → 如竞品403/超时 → 标记跳过，继续其他竞品

Step 4: 内容差距分析
        → content-tracker.list_content(client_id)     // 我方已有内容
        → 对比竞品覆盖 vs 我方覆盖：
          - 竞品有但我方没有的主题 → 🔴 内容缺口
          - 我方有但竞品没有的主题 → 🟢 差异化优势
          - 双方都有但我方更弱的主题 → 🟡 需加强
        → 按商业价值排序（设备类 > 原料类 > 科普类）

Step 5: 质量验证（不可跳过）
        → 抓取成功率检查：≥60%竞品成功抓取才算有效
        → 内容差距覆盖度：至少扫描每个竞品的博客/产品/FAQ三个板块
        → 交叉验证：fetch结果与seo-checker结果是否自洽（如竞品声称有Schema但check_structured_data为空 → 标记异常）

Step 6: 输出+记录
        → 保存到 客户-XX/数据/YYYY-MM_竞品动态简报.md
        → client-manager.add_timeline(id, "竞品分析完成: X个竞品, X个内容缺口, X个差异化优势")
```

## MCP 调用链：快速竞品扫描（daily-ops自动调用）

```
快速模式仅执行 Step 3（抓取博客页），对比上次抓取：
→ 竞品有新文章 → 🟡 标记需48h内响应
→ 竞品无变化 → ✅ 跳过
→ 输出一行摘要到巡检日报
```

## 竞品分析维度

对每个竞品网站分析：

| 维度 | 检查内容 | 数据来源 |
|------|----------|---------|
| 内容数量 | 博客总数、近30天发布频率 | fetch(blog页) |
| 内容深度 | 平均字数、表格/FAQ使用、多媒体 | fetch(文章页抽检) |
| 语言覆盖 | 多语言版本数量 | fetch(首页) |
| 技术水平 | SSL、响应时间、Schema数量 | seo-checker |
| 内容差距 | 竞品覆盖但我方缺失的主题 | 对比content-tracker |
| 我方优势 | 我方有但竞品没有的差异化 | 对比content-tracker |

## 与其他技能联动

| 发现 | 触发技能 | 触发方式 |
|------|---------|---------|
| 竞品48h内发新文章 | → **daily-growth** 维度7竞品响应 | 每日巡检自动标记 |
