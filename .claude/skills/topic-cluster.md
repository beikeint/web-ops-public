# 内容集群路线图技能

> 触发指令：`[客户名] 规划内容集群` / `[客户名] 内容路线图`  
> 执行频率：首次启用时全量规划，此后每季度更新  
> 人工确认：⏸️ 集群方案需确认后作为后续选题的基础

---

## 什么是 Topic Cluster

```
            ┌─── 子文章A（长尾词1）
            │
支柱页面 ───┼─── 子文章B（长尾词2）
（核心主题） │
            ├─── 子文章C（长尾词3）
            │
            └─── 子文章D（长尾词4）

所有子文章 ←内链→ 支柱页面
子文章之间 ←内链→ 相互连接
```

**为什么重要**：
- Google通过内链关系理解"主题权威性"
- 一个完整的Topic Cluster比10篇零散文章的排名效果好3-5倍
- 避免关键词自我竞争（多篇文章争同一个词）
- 让每月选题不再拍脑袋，而是按路线图推进

## MCP 调用链

```
Step 1: 加载客户信息
        → client-manager.get_client(client_id)
        → 获取：行业、产品线、目标市场、已发布内容

Step 2: 盘点已有内容
        → content-tracker.list_content(client_id)
        → fetch(域名/sitemap-index.xml) → 提取所有已有页面
        → 分类：产品页 / 博客 / 着陆页 / 其他

Step 3: 竞品内容版图
        → fetch(竞品博客页) × N
        → 提取竞品的内容主题分布
        → 识别竞品的Topic Cluster结构

Step 4: 关键词宇宙构建
        → 基于产品线、买家意图、行业术语，构建关键词宇宙
        → 分层：核心词（Head）→ 中频词（Body）→ 长尾词（Tail）
        → 按搜索意图分类：信息型 / 商业型 / 导航型 / 交易型

Step 5: 设计 Topic Cluster
        → 确定3-5个支柱主题
        → 每个支柱下规划6-12篇子文章
        → 设计内链矩阵（谁链接谁）
        → 标注优先级和时间线

Step 6: 输出路线图
        → 保存到 客户-XX/数据/内容集群路线图.md
        → client-manager.add_timeline(id, "完成内容集群规划: X个支柱/X篇子文章")
```

**⏸️ 暂停**：输出后等确认。确认后作为后续每月选题的基础。

## 支柱主题选择标准

| 维度 | 标准 |
|------|------|
| 搜索量 | 核心词月搜索量>1000 |
| 商业价值 | 与客户产品/服务直接相关 |
| 竞争度 | 不选竞争白热化的词，选可切入的 |
| 内容可扩展性 | 至少能拆出6篇子文章 |
| 已有内容覆盖 | 优先选已有部分内容可复用的主题 |

## 子文章设计原则

1. **一篇一词**：每篇子文章锁定一个主关键词，不重复
2. **意图覆盖全**：认知→考虑→决策阶段都要有文章
3. **内链必达**：每篇子文章至少链回支柱页1次 + 链接1篇相关子文章
4. **产品自然导入**：决策阶段文章自然链接到产品页
5. **字数递进**：支柱页5000+，对比指南3000+，应用场景2000+，FAQ 1500+

## 质量检查清单（输出前逐项确认）

- [ ] 每个支柱主题的搜索量预估合理（不是凭空想象）
- [ ] 支柱页关键词与子文章关键词无蚕食冲突
- [ ] 每篇子文章都有明确的搜索意图分类（认知/考虑/决策）
- [ ] 内链矩阵中每篇子文章至少链回支柱页1次
- [ ] 3个买家阶段（认知/考虑/决策）都有覆盖，不偏科
- [ ] 时间线可执行（每双周1篇，总量不超过实际产能）
- [ ] 已与 content-tracker 现有内容对比，不重复规划已有文章
- [ ] timeline已记录：client-manager.add_timeline(id, "集群规划完成: X个支柱+X篇子文章")

## 更新机制

- **每月**：标记已完成文章，调整优先级
- **每季度**：全量Review——新增主题？删除不再相关的？调整时间线？
- **竞品变动时**：竞品在某个集群方向发力 → 提升该集群优先级

## 批准后流程

集群方案确认后，自动执行：
1. 将子文章列表输入 `data-analysis` 作为每月选题池
2. 每次 `content-production` 写作时自动读取集群内链设计
3. 每次 `content-refresh` 体检时按集群维度评估完整度
