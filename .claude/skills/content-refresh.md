# 内容衰退检测与更新技能

> 触发指令：`[客户名] 内容体检` / `[客户名] 更新旧文章：[slug]`  
> 执行频率：每月1次全量体检，发现衰退即更新  
> 人工确认：⏸️ 全量体检的更新方案需确认后批量执行；单篇更新自动执行  
> 核心原则：**体检不是目的，更新才是结果。报告必须以"已修复X篇"结束，不允许以"待办清单"结束。**

---

## 模式一：全量内容体检（`[客户名] 内容体检`）

### MCP 调用链

```
Step 1: 加载客户
        → client-manager.get_client(client_id)
        → 获取域名、语言列表

Step 2: 获取已发布内容清单
        → content-tracker.list_content(client_id, status="published")
        → 提取：标题、发布日期、URL、关键词

Step 3: 逐篇检测（并行，每篇一个fetch）
        → fetch(文章URL) → 提取：
          - 正文字数
          - H2/H3结构
          - 内链数量
          - 图片数量及alt标签
          - FAQ数量
          - 是否有Schema标记
          - 是否有答案胶囊（首段是否直接回答问题）
          - 最后更新日期（如页面有显示）

Step 4: 衰退评分（每篇文章 100 分）

    时效性（30分）
    ├─ 发布<30天：30
    ├─ 30-90天：20
    ├─ 90-180天：10
    ├─ 180-365天：5
    └─ >365天：0

    内容深度（25分）
    ├─ 字数>3000：10 / 2000-3000：7 / <2000：3
    ├─ 表格>=2：5 / 1：3 / 0：0
    ├─ 数据点>=5：5 / 3-4：3 / <3：0
    └─ FAQ>=5：5 / 3-4：3 / <3：0

    SEO完整度（25分）
    ├─ 内链>=3：10 / 1-2：5 / 0：0
    ├─ 图片均有alt：5 / 部分有：3 / 无图或无alt：0
    ├─ 有Schema标记：5 / 无：0
    └─ 有答案胶囊：5 / 无：0

    GEO适配（20分）
    ├─ 有FAQ Schema：5 / 无：0
    ├─ 段落长度<=120词：5 / 有超长段落：0
    ├─ 有表格呈现数据：5 / 无：0
    └─ 数据有来源标注：5 / 无：0

Step 5: 分类 + 立即行动

    🔴 评分<40 → **立即修复**（不等确认）
    → FAQ缺失 → 当场补5个FAQ + Schema
    → 无内链 → 当场补3-5个内链
    → 无Schema → 当场补 Article + FAQPage Schema
    → 数据过时 → 更新数据，保留结构
    → 修复后重新评分，确认分数提升到≥40

    🟡 评分40-65 → ⏸️ 输出更新方案，确认后批量执行
    → 更新方案列出具体修改项和预期分数提升

    🟢 评分>65 → 短期内无需处理

Step 6: 验证闭环
        → 所有🔴修复完成后：npm run build → 确认无报错
        → 重新评分确认所有🔴文章≥40分
        → 如有部署：site-monitor.check_site → 验证上线

Step 7: 记录
        → 保存到 客户-XX/数据/YYYY-MM_内容体检报告.md
        → client-manager.add_timeline(id, "内容体检: X篇已修复, X篇待确认更新")
        → content-tracker.update_content(...) 更新每篇文章状态
```

### 内容体检报告格式

```
# [客户名] 内容体检报告 — YYYY年MM月

## 总览
- 已发布文章：X篇
- 🔴 需紧急更新：X篇
- 🟡 建议更新：X篇
- 🟢 状态良好：X篇
- 平均内容评分：XX/100

## 详细评分

| # | 标题 | 发布日期 | 天数 | 字数 | 内链 | FAQ | Schema | 评分 | 状态 |
|---|------|---------|------|------|------|-----|--------|------|------|
| 1 | xxx | 2026-03 | 38天 | 3200 | 5 | 7 | ✅ | 82 | 🟢 |
| 2 | xxx | 2025-10 | 178天 | 1800 | 1 | 0 | ❌ | 28 | 🔴 |

## ✅ 已修复（🔴评分<40的文章，当场修复）

### [文章标题]
- 修复前评分：XX → 修复后评分：XX
- 修复内容：补FAQ X个 / 补内链 X个 / 补Schema / 更新数据
- 验证：构建通过 ✅ / 评分提升到≥40 ✅

## ⏸️ 待确认更新（🟡评分40-65的文章）

### [文章标题]
- 当前评分：XX/100
- 主要扣分项：[时效性 / 字数不足 / 无FAQ / 无内链]
- 更新方案：
  - [ ] 补充XXX内容（+约500词）
  - [ ] 添加FAQ X个问题
  - [ ] 添加内链到 [产品页/其他博客]
- 预估分数提升：XX → XX

**确认后执行 `[客户名] 更新旧文章：[slug]` 逐篇更新**

## 下次体检建议：YYYY-MM-DD
```

## 模式二：单篇更新（`[客户名] 更新旧文章：[slug]`）

### MCP 调用链

```
Step 1: 加载客户 + 文章信息
        → client-manager.get_client(client_id)
        → fetch(文章URL) → 获取当前内容状态

Step 2: 诊断问题
        → 运行衰退评分
        → 对比竞品同主题最新文章（fetch竞品文章）
        → 识别：过时数据 / 缺失章节 / 新的行业变化

Step 3: 制定更新方案（⏸️ 需确认）
        → 哪些内容保留
        → 哪些内容需要更新/重写
        → 哪些内容需要新增
        → 更新后预期评分提升

Step 4: 执行更新（确认后）
        → 更新文章内容（在blog-posts.ts中修改对应body）
        → 更新日期
        → 多语言同步更新
        → 添加/更新Schema
        → 补充内链

Step 5: 质量验证
        → 重新运行衰退评分，确认分数提升
        → 运行content-production的阶段6质量评分
        → 构建验证

Step 6: 记录
        → content-tracker.update_content(...)
        → client-manager.add_timeline(id, "更新文章: [标题]")
        → 保存到 客户-XX/内容/YYYY-MM_更新_[slug].md
```

## 更新策略

### 哪些文章优先更新

| 优先级 | 条件 | 理由 |
|--------|------|------|
| P0 | 排名在第2-3页且流量在下降 | 投入最小回报最大，可能重回第1页 |
| P1 | 发布>180天且评分<40 | 严重拖累整站质量 |
| P2 | 竞品在同主题发了新文章 | 不更新会被超越 |
| P3 | 字数<2000且无FAQ | 快速补充即可提升 |

### 更新不等于重写

| 情况 | 动作 |
|------|------|
| 数据过时 | 更新数据，保留结构 |
| 缺少FAQ | 添加5-7个FAQ |
| 缺少内链 | 补充3-5个内链 |
| 缺少Schema | 添加Article+FAQ Schema |
| 字数不足 | 补充1-2个H2章节 |
| 整体过时 | 保留框架，重写70%以上内容 |

| 发现 | 联动 |
|------|------|
